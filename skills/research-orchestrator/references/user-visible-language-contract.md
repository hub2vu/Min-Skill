User-Visible Language Contract

User-facing research output must read like a research progress report, not like a system panel.

Never expose these engine-facing or orchestration-facing terms in the public response:

- `R1` through `R9`
- `current stage`
- `last locked`
- `blocker`
- `next advance condition`
- `transition/reason`
- `capsule`
- `manuscript-entry`
- `current-run.md`
- `status.md`
- raw internal state objects
- operating contract
- research contract archive
- archive contract
- operating archive
- knowledge archive
- contract file
- skill file
- bootstrapping
- workflow contract confirmation
- archive unpacked because of rules
- "I reread AGENTS" style orchestration narration
- "after checking the attached archive" style operating narration
- "after confirming the contract files" style operating narration

Use standard research-process language instead, in the active user language when practical. Preferred visible labels are:

- research question refinement
- concept and scope definition
- prior-art review
- evidence appraisal
- competing explanation comparison
- argument formation
- section drafting
- integration and revision
- final completion readiness

Every substantive public research response should naturally cover:

- what was examined
- what was learned
- what is now stable enough to rely on
- what this turn narrowed or newly opened when relevant
- why the work is staying here, moving forward, or returning earlier
- what remains insufficient
- how the argument or writing readiness changed
- what the next justified move is
- the latest cumulative ZIP path or attachment

The first public-facing paragraph should talk about the research work itself, not about hidden preparation steps. Open with what was examined, what shifted, or what remains unresolved. Do not open with archive, contract, skill, or bootstrap narration unless the user is explicitly auditing the system.

Do not use:

- percentages
- progress bars
- system state tables
- terse status headers that read like diagnostics
- raw file names as the main visible story
- archive-governance language as if the user were auditing the engine instead of receiving a research update

Good public language explains judgment, not machine state. The user should understand why the research is where it is without learning the engine's internal vocabulary.

The same language rule applies to user-facing archive prose. Public archive guides, status notes, turn notes, section files, and similar user-visible documents should avoid system-meta wording and default to the user's language when practical.

Public-facing metadata labels and update-log verbs should also default to the user's language when practical. Do not leave English field labels or log verbs as the unexamined default in a non-English user session when user-language labels are practical.

Prefer natural phrasing such as:

- "I checked whether the earlier conclusion still holds in the Korean material."
- "That narrows the argument rather than replacing it."
- "A bounded section draft is now justified, but a full manuscript is still not ready."

Keep those ideas in plain prose rather than rendering them as raw status fields.
